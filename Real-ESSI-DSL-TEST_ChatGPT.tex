\section{Introduction}

This   chapter   presents   the  domain  specific  language  developed  for  the
Real-ESSI. The language was
designed  with a primary goal of developing FEA models and interfacing them with
various   Real-ESSI   functionalities.  In  addition  to  that,  syntax  is  used  to
self-document models, provide physical-unit safety,
provide common flow control structures, provide modularity to scripting via user
functions  and  ``include'' files, and provide an interactive environment within
which models can be created, validated and verified.


The  development of Real-ESSI Domain specific language (DSL) (the Finite Element
Interpreter,    \FEI{})    is   based   on   LEX   \citep{Lesk1975}   and   YACC
\citep{Johnson1975}.

Self-documenting  ensures  that  the  resulting  model  script  is  readable and
understandable with little or no reference to the users
manual.  This  is  accomplished  by  providing  a  command grammar structure and
wording  similar  to what would be used in a natural language description of the
problem.

FEA  analysis  is  unitless,  that  is, all calculations are carried out without
referencing  a  particular unit system. This leaves the task of unit correctness
up  to  the  user  of  FEA  analysis. This represents a recurring source of error in FEA
analysis.  Physical  unit  safety  is  enforced in Real-ESSI by implementing all base
variables  as physical quantities, that is, all variables have a unit associated
with  it.  The adimensional unit is the base unit for those variables which have
no  relevant unit (like node numbers). Command calls are sensitive to units. For
example, the node creation command call expects the node coordinates to be input
with    the    corresponding  units  (length  in  this  case). Additionally, the
programming/command language naturally
supports  operation  with  units  like  arithmetic  operations (quantities with
different  unit  types will not add or subtract but may be multiplied).
This  approach  to  FEA  with  unit  awareness  provides  an additional layer of
security  to  FEA  calculations,  and  forces  the user to carefully think about
units. This can help catch some common mistakes.


The   Real-ESSI    language  provides  modularity  through  the  \lstinline|include|
directive/command, and user
functions.
%\footnote{user  functions  have  not been implemented yet}. 
This allows
complex  analysis  cases to be parameterized into modules and functions which can
be reused in other models.


Finally,  an  emphasis  is  placed on model verification and validation. To this
end, Real-ESSI  provides  an interactive programming environment with all
 the    ESSI  syntax available.  By using this environment. the user can develop
 tests to detect errors in the
model  that  are  not programming errors. For example, the user can query nodes
and  elements  to  see  if  they  are  set  to appropriate states. Also, several
standard tools are provided to check element validity (Jacobian, etc.).



The  ESSI  language  provides  reduced  model  development time by providing the
aforementioned  features  along  with  meaningful error reporting (of syntax and
grammatical  errors),  a  help  system,  command completion and highlighting for
several open source and commercial text editors.


Some  additional  ideas are given by \cite{Dmitriev2004}, \cite{Stroustrup2005},
\cite{Niebler2005}, \cite{Mernik2005}, \cite{Ward2003}, etc.

\section{Domain  Specific  Language (DSL), English Language Binding}
\label{DSL_english}

Overview of the language syntax.

\begin{itemize}

  
  \item Each command line has to end with a semicolon "\texttt{;}"

  \item  Comment on a line begins with either "\texttt{//}" or "\texttt{!}" and last
  until the end of current line.

  \item  Units are required (see more below) for all quantities and variables.

  \item  Include statements allow splitting source into several files

  \item  All  variables  are  double precision (i.e. floats)  with  a  unit  attached.

  \item  All  standard  arithmetic  operations  are  implemented,  and  are unit
  sensitive.

  \item  Internally, all units are represented in the base SI units ($m$ - $s$ -
  $kg$).

  \item  The  syntax  ignores  extra  white  spaces,  tabulations  and newlines.
  Wherever  they  appear,  they are there for code readability only. (This is why
  all commands need to end with a semicolon).
  
  \item The user should be familiar with the list of the reserved keywords from
  Section~\ref{section:list-of-reserved-keywords} on page~\pageref{section:list-of-reserved-keywords}.


\end{itemize}

\subsection{Real-ESSI Variables, Basic Units and Flow Control}


Variables are defined using the assignment (\texttt{=}) operator. For example,

\begin{lstlisting}
 var_x   =  7;      //Results in the variable x be set to 7 (unitless)
 var_y = 3.972e+2;  //Scientific notation is available.
\end{lstlisting}

The   language   contains   a   list   of  reserved  keywords.  Throughout  this
documentation,  reserved  keywords  are  highlighted  in  {\color{blue}blue}  or
{\color[rgb]{0.4,0,0}red}.

All  standard  arithmetic  operations  are  available  between  variables. These
operations can be combined arbitrarily and grouped together with parentheses.

\begin{lstlisting}
var_a = var_x + var_y;      // Addition
var_b = var_x - var_y;      // Subtraction
var_c = var_x * var_y;        // Product
var_d = var_x / var_y;        // Quotient
var_e = var_y % var_x;        // Modulus (how many times x fits in y)
\end{lstlisting}

The '\lstinline|print|' command can be used to display the current value of a variable.

\begin{lstlisting}
print var_x;
print var_y;
print var_a;
print var_b;
print var_c;
print var_d;
print var_e;
\end{lstlisting}

\begin{lstlisting}[style=commandline]
var_x = 7 []
var_y = 397.2 []
var_a = 404.2 []
var_b = -390.2 []
var_c = 2780.4 []
var_d = 0.0176234 []
var_e = 5.2 []
\end{lstlisting}

Here  the  ``unit'' (sign) \lstinline|[]|  means  that  the  quantities are unitless. 


The command
'\lstinline|whos|'  is used to see all the currently defined variables and their
values.    After  a  fresh  start of essi, needed to clear up all the previously
defined  variables,  command  \lstinline|whos;|'  produces a list of predefined
variables:

\noindent
Predefined  variables  shown  above  have  a preceding asterisk to show they are
locked variables which cannot be modified. The purpose of these locked variables
are  to  provide  names  for  units.  Imperial units are also supported as shown
above.

The units for variable are shown between  the  brackets. Note that unit
variables  have  the  same  name  as  their unit, which is not the case for user
defined  variables.  Variables  preceded  by  a star (\lstinline|*|) are locked
variables which can't be modified.

For  example,  the  variable '\lstinline|m|' defines 'meter'. So to define a
new variable \lstinline|L1| which has meter units we do:

\begin{lstlisting}
L1 = 1*m;     //  Defines L1 to 1 m.
L2 = 40*mm;   //  Defines L2 to be 40 millimeters.
\end{lstlisting}

Even  though  L2  was created with millimeter units, it is stored in base units.

\lstinline|print L2;| displays

\begin{lstlisting}[style=commandline]
 L2 = 0.04 [m]
\end{lstlisting}



As additional examples, let us define few forces:

\begin{lstlisting}
F1 = 10*kN;
F2 = 300*N;
F3 = 4*kg*g;
\end{lstlisting}

Here \lstinline|g| is the predefined acceleration due to gravity.

Arithmetic operations do check (and enforce) for unit consistency.  For  example, \lstinline|foo = L1 + F1;|
produces  an  error because units are not compatible. However,
\lstinline|bar = L1 + L2;| is acceptable. On the other hand, multiplication,
division and modulus,
always  work  because the result produces a quantity with new units (except when
the adimensional quantity is involved).

\begin{lstlisting}
A = L1*L2;
Stress_n = F1 / A;
\end{lstlisting}

Units  for all variables are internally converted to SI units ($kg - m - s$) and
stored in that unit system.
Variables   can   be   \emph{displayed}  using  different  units  by  using  the
\lstinline|[]| operator. This does not change the variable, it just displays the
value of variable with required unit. For example,

\begin{lstlisting}
print Stress_n;           //Print in base SI units.
print Stress_n in Pa;       //Print in Pascal
print Stress_n in kPa;      //Print in kilo Pascal
\end{lstlisting}

\begin{lstlisting}[style=commandline]
   Stress_n = 250000 [kg*m^-1*s^-2]

   Stress_n = 250000 [Pa]

   Stress_n = 250 [kPa]
\end{lstlisting}

The DSL  provides  functions  to test the physical units of variables. For
example,

\lstinline|print isForce(F1);|  \hspace*{1cm}

Will  print  an  adimensional, Boolean  \lstinline|1| because \lstinline|F1| has units of
force. While,

\lstinline|print isPressure(F);|  \hspace*{1cm}

will    print    an    adimensional, Boolean   \lstinline|0|.  The  language also provides
comparison of
quantities with same units (remember all values are compared in SI Units).

\lstinline|print F1 > F2;|  \hspace*{1cm}

will  print  an  adimensional, Boolean  \lstinline|1| since \lstinline|F1| is greater than
\lstinline|F2|.

The  program  flow  can  be controlled with \lstinline|if| and \lstinline|while|
statements, i.e.:
%
\begin{lstlisting}
if (isForce(F1))
{
 print F1;    // This will be executed
};

if (isForce(L1))
{
 print L1;    // This will not.
};
\end{lstlisting}
%
Note the necessary semicolon (\lstinline|;|) at the closing brace. Unlike C/C++,
the braces are always necessary. Closing colon is also always necessary.

The ``else'' statement is also available:
%
\begin{lstlisting}
if (isForce(L1))
{
 print L1;    // This will not execute
}
else
{
 print L2;    // This will execute instead
};
\end{lstlisting}



While loops are also available:

\begin{lstlisting}
i = 0;
while( i < 10)
{
 print i;
 i = i +1;
};
\end{lstlisting}

\subsection{Modeling}


%\noindent  \textbf{Note:}
This section details ESSI modeling commands.
 Angle  brackets  \lstinline|<>| are used for quantity  or variable placeholder,
 that is, they indicate where user input goes.
Within      the   angle   brackets,   the   expected   unit   type  is  given as
well,  i.e..
\lstinline|<L>|    means  the  command  expects an input with a value and a
length unit. The symbol \lstinline|<.>| represents the adimensional quantity.

In  addition  to that,  the  vertical  bar  \lstinline+|+ (``OR'' sign)) is used
to separate two or more keyword
options,   i.e.   \lstinline+[a|b|c]+   is   used   indicate   keyword   options
\lstinline+a+ or \lstinline+b+ or \lstinline+c+. The symbol \lstinline+|...|+ is
used  to  denote  where several long options exist and are explained elsewhere
(an example of this is available below in  a material model definitions).

All  commands  require  unit  consistency.  Base units, SI  or  other  can be used as
indicated below:
\begin{itemize}
  \item  length, symbol  $L$, units [m, inch, ft]
  \item  mass, symbol    $M$, units [kg, lbm],
  \item  time, symbol    $T$, units [s]
\end{itemize}

Derived units can also be used:

\begin{itemize}
  \item angle, symbol rad (radian), unit [$dimensionless, L/L$]
  \item force, symbol N (Newton), units [$N, kN, MN, M*L/T^2$],
  \item stress, symbol Pa (Pascal), units [$Pa, kPa, MPa, N/L^2, M/L/T^2$]
  \item strain, symbol (no symbol), units [$L/L$]
  \item mass density, symbol (no symbol), units [$M/L^3$]
  \item force density, symbol (no symbol), units [$M/L^2/T^2$]
\end{itemize}


All models have to be named: \lstinline|model name "model_name_string";| This is
important as output files are named based on model name.

Each loading stage has to be named as well.

page~\pageref{LSII} in \citet{Jeremic2004d}.} is defined like this:

\lstinline|new loading stage "loading stage name string";|

In  addition  to model name, loading stage name is used for output file name for
given loading stage.

% temp-out % The command is
% temp-out %
% temp-out % \begin{lstlisting}
% temp-out % add material # <.> type Pisano
% temp-out %             mass_density '=' exp
% temp-out %             elastic_modulus_1atm '=' exp
% temp-out %             poisson_ratio '=' exp
% temp-out %             M_in '=' exp
% temp-out %             kd_in '=' exp
% temp-out %             xi_in '=' exp
% temp-out %             h_in '=' exp
% temp-out %             m_in '=' exp
% temp-out %             initial_confining_stress '=' exp
% temp-out %             n_in '=' exp
% temp-out %             a_in '=' exp
% temp-out %             eplcum_cr_in '=' exp
% temp-out % \end{lstlisting}
% temp-out 
% temp-out % where
% temp-out 
% temp-out %    \begin{itemize}
% temp-out %      \item \verb|mass_density|  Density of the material.
% temp-out %      \item \verb|elastic_modulus_1atm|  Elastic modulus of the material at $1\,\mathrm{atm}$ of pressure.
% temp-out %      \item \verb|poisson_ratio|
% temp-out %      \item \verb|M_in|  Critical stress ratio in $p$-$q$ space, for triaxial compression.
% temp-out %      \item \verb|kd_in| Parameter controlling dilatancy.
% temp-out %      \item \verb|xi_in| Parameter controlling dilatancy.
% temp-out %      \item \verb|h_in| Hardening parameter
% temp-out %      \item \verb|m_in|  Hardening parameter
% temp-out %      \item \verb|initial_confining_stress| Confining stress for the first step.
% temp-out %      \item \verb|n_in|  Parameter controlling the shape of the deviatoric section of the boundary surface.
% temp-out %      \item \verb|a_in|  Exponent for pressure dependent elastic moduli
% temp-out %      \item  \verb|eplcum_cr_in|    Critical cumulated plastic strain upon stress
% temp-out %      reversal (for fixing the overshooting problem)
% temp-out %    \end{itemize}
% temp-out

% temp-out \section{Checking the Model}
% temp-out \label{check_model}
% temp-out 
% temp-out Real-ESSI provides model check capability:
% temp-out 
% temp-out \begin{lstlisting}
% temp-out  check model ;
% temp-out \end{lstlisting}
% temp-out 
% temp-out This command will cycle  over all the domain components, including Nodes,
% temp-out Elements, Loads, Constraints, etc. 
% temp-out and execute the \texttt{checkModel()} function for each. Each domain component
% temp-out writes/reports
% temp-out to the terminal and to 
% temp-out the \textit{essi.log} file, if an error is found.
% temp-out For example, bricks will report when the computed Jacobian is negative 
% temp-out  and other similar errors. Nodes that are not connected will be reported as well.
% temp-out %
% temp-out If the diagnostic log is empty, it means that the mesh has passed all tests.
% temp-out %
% temp-out Additionally, an output HDF5 file is produced that can be used to display the
% temp-out mesh and do further visual inspections of the model. This file will have initial
% temp-out conditions as outputs for element sand nodes.
% temp-out 
% temp-out 
% temp-out %
% temp-out %Besides, this command will check if any defined node is never used.
% temp-out %If any defined node is never used, essi will report the node number and exit.
% temp-out %
% temp-out Command  \lstinline|check model;|  represents a dry  run through the model that is used 
% temp-out to check the model before a full analysis.
% temp-out %
% temp-out Model check is highly recommended before initial stages of a full analysis are
% temp-out executed.

% temp-out \section{Constitutive Testing}
% temp-out \label{constitutive_testing}
% temp-out 
% temp-out 
% temp-out Material models can be tested using constitutive drivers which exercise single
% temp-out material models. RealE-SSI implements two such drivers.
% temp-out 
% temp-out \begin{enumerate}
% temp-out   \item Bardet Driver. Bardet-type constraints can be used to simulate conditions such as drained or undrained triaxial testing with strain or stress control or direct shear testing with shear control. 
% temp-out   \item Direct Strain Driver. This driver applies a given strain history (specified by the user) to a material model. 
% temp-out \end{enumerate}
% temp-out 
% temp-out Both these drivers produce identical output: the files \texttt{Stress.feioutput} and \texttt{Strain.feioutput} which contain stress and strain tensor components at each step. Additionally, the drivers may print out material internal information to the file \texttt{Material\_Output.feioutput}. For the stress and strain files, each line of these files contain the stresses and strains organized in the following manner:
% temp-out 
% temp-out \noindent \texttt{Stress.feioutput} $\rightarrow$ $\sigma_{11}$ $\sigma_{22}$ $\sigma_{33}$ $\sigma_{12}$ $\sigma_{13}$ $\sigma_{23} $.
% temp-out 
% temp-out \noindent \texttt{Strain.feioutput} $\rightarrow$ $\epsilon_{11}$ $\epsilon_{22}$ $\epsilon_{33}$ $\epsilon_{12}$ $\epsilon_{13}$ $\epsilon_{23} $.
% temp-out 
% temp-out The Bardet driver has the following format. 
% temp-out 
% temp-out \begin{lstlisting}
% temp-out   simulate constitutive testing BARDETMETHOD use material # <.> 
% temp-out     scale_factor = <.>  
% temp-out     series_file = <string>  
% temp-out     sigma0 = ( <F/L^2> , <F/L^2> , <F/L^2> , <F/L^2> , <F/L^2> , <F/L^2> )
% temp-out     verbose_output = <.>
% temp-out \end{lstlisting}
% temp-out 
% temp-out Where,
% temp-out 
% temp-out \begin{itemize}
% temp-out   \item \lstinline{BARDETMETHOD} can have any one of the following values:
% temp-out   \begin{itemize}
% temp-out     \item \lstinline{CONSTANT_P_TRIAXIAL_LOADING_STRAIN_CONTROL}: Triaxial loading with $p$ kept constant. In this case the input file is interpreted as strain increments in the $\epsilon_{11}$ component. 
% temp-out     \item \lstinline{DRAINED_TRIAXIAL_LOADING_STRESS_CONTROL}: Drained Triaxial loading. In this case the input file is interpreted as stress increments in the $\sigma_{11}$ component. 
% temp-out     \item \lstinline{DRAINED_TRIAXIAL_LOADING_STRAIN_CONTROL}: Drained Triaxial loading. In this case the input file is interpreted as strain increments in the $\epsilon_{11}$ component. 
% temp-out     \item \lstinline{UNDRAINED_TRIAXIAL_LOADING_STRAIN_CONTROL}: Undrained Triaxial loading. In this case the input file is interpreted as strain increments in the $\epsilon_{11}$ component. 
% temp-out     \item \lstinline{UNDRAINED_TRIAXIAL_LOADING_STRESS_CONTROL}: Undrained Triaxial loading. In this case the input file is interpreted as stress increments in the $\sigma_{11}$ component.
% temp-out     \item \lstinline{UNDRAINED_SIMPLE_SHEAR_LOADING_STRAIN_CONTROL}: Undrained simple-shear loading. In this case the input file is interpreted as angular strain increments in the $\gamma_{12} = 2\epsilon_{12}$ component. 
% temp-out   \end{itemize}
% temp-out   \item \lstinline{scale_factor}: Can be used to scale the series file arbitrarily. 
% temp-out   \item \lstinline{series_file}: String specifying the path to the file containing the increments (might be interpreted as strain or stress depending on the method chosen). Each line of the file contains one increment. 
% temp-out   \item \lstinline{sigma0}: Components of the initial stress for the material, given in the order: $(\sigma_{11},\,\sigma_{22},\,\sigma_{33},\,\sigma_{12},\,\sigma_{13},\,\sigma_{23} )$.
% temp-out   \item \lstinline{verbose_output}(\texttt{=N}) Whether the driver should print extra information about the material model every $N$ steps. If Takes value $0$ (no output) or $N$ (do output every $N$ increments). Each material implements its own output, so the format of the \texttt{Material\_Output.feioutput} file is variable and material dependent. 
% temp-out \end{itemize}
% temp-out 
% temp-out The direct strain driver has the following format. 
% temp-out 
% temp-out \begin{lstlisting}
% temp-out   simulate constitutive testing DIRECT_STRAIN use material # <.> 
% temp-out     scale_factor = <.>  
% temp-out     series_file = <string>  
% temp-out     sigma0 = ( <F/L^2> , <F/L^2> , <F/L^2> , <F/L^2> , <F/L^2> , <F/L^2> )
% temp-out     verbose_output = <.>
% temp-out \end{lstlisting}
% temp-out 
% temp-out Where al the arguments are the same as the Bardet driver. In this case each line of the file contains all six components of the strain increment to be applied. For example:
% temp-out 
% temp-out \noindent \lstinline{series_file = "increments.txt"} where each line in \texttt{increments.txt} contains  $\mathrm{d}\epsilon_{11}$ $\mathrm{d}\epsilon_{22}$ $\mathrm{d}\epsilon_{33}$ $\mathrm{d}\epsilon_{12}$ $\mathrm{d}\epsilon_{13}$ $\mathrm{d}\epsilon_{23} $.
% temp-out 
% temp-out % 
% temp-out % 
% temp-out % 
% temp-out % \begin{itemize}
% temp-out %   \item \lstinline|strain_increment_size|: increment size of the strain
% temp-out %   \item \lstinline|maximum_strain|: maximum strain
% temp-out %   \item \lstinline|number_of_times_reaching_maximum_strain|: number of times which the current strain
% temp-out %                    reaches the maximum strain (either in positive or negative). The following figures
% temp-out %                    show the case which number\_of\_times\_reaching\_maximum\_strain is set to 1, 2, and 3.
% temp-out % \end{itemize}
% temp-out % 
% temp-out % 
% temp-out % 
% temp-out % 
% temp-out % \begin{figure}[!htbp]
% temp-out % \begin{center}
% temp-out % \subfigure[number\_of\_times\_reaching\_maximum\_strain=1]{\label{fig:stefan23}\includegraphics[scale=0.27]{../../Figure-files/_Chapter_SoftwareHardware_Domain_Specific_Language_English/number_of_times_reaching_maximum_strain_1.pdf}}
% temp-out % \subfigure[number\_of\_times\_reaching\_maximum\_strain=2]{\label{fig:stefan24}\includegraphics[scale=0.27]{../../Figure-files/_Chapter_SoftwareHardware_Domain_Specific_Language_English/number_of_times_reaching_maximum_strain_2.pdf}}
% temp-out % \subfigure[number\_of\_times\_reaching\_maximum\_strain=3]{\label{fig:ste24-23}\includegraphics[scale=0.27]{../../Figure-files/_Chapter_SoftwareHardware_Domain_Specific_Language_English/number_of_times_reaching_maximum_strain_3.pdf}}
% temp-out % \end{center}
% temp-out % \caption{Number of times reaching maximum strain in constitutive testing}
% temp-out % \label{fig:stefanResidual}
% temp-out % \end{figure}
% temp-out %

% temp-out % number_of_times_reaching_maximum_strain = <.>;
% temp-out % \end{lstlisting}
% temp-out % 
% temp-out % 
% temp-out % \begin{lstlisting}
% temp-out % simulate constitutive testing undrained triaxial
% temp-out % use material # <.>
% temp-out % strain_increment_size = <.>
% temp-out % maximum_strain = <.>
% temp-out % number_of_times_reaching_maximum_strain = <.>;
% temp-out % \end{lstlisting}
% temp-out % 
% temp-out % 
% temp-out % \begin{lstlisting}
% temp-out % simulate constitutive testing undrained simple shear
% temp-out % use material # <.>
% temp-out % strain_increment_size = <.>
% temp-out % maximum_strain = <.>
% temp-out % number_of_times_reaching_maximum_strain = <.>;
% temp-out % \end{lstlisting}
% temp-out % 
% temp-out % 
% temp-out % \begin{lstlisting}
% temp-out % simulate constitutive testing undrained triaxial
% temp-out % use material # <.>
% temp-out % strain_increment_size = <.>
% temp-out % maximum_strain = <.>
% temp-out % number_of_times_reaching_maximum_strain = <.>;
% temp-out % \end{lstlisting}
% temp-out % 
% temp-out % 
% temp-out % The output files will be Results.out, Strain.out, and Stress.out with the following contents:
% temp-out % \begin{itemize}
% temp-out % \item Results.feioutput: $-\epsilon_{11} , -\epsilon_{12} , -\epsilon_{22} , -\epsilon_{33} , -\epsilon_{volumetric} , -\epsilon_{deviatoric} , p , q(\sigma_{22} - \sigma_{11}) , q$
% temp-out % \item Strain.feioutput : $-\epsilon_{11} , -\epsilon_{12} , -\epsilon_{33} , -\epsilon_{33} , -\epsilon_{volumetric} ,  -\epsilon_{deviatoric}$

% temp-out % \item pq.feioutput     : p , q (for all the start stress, cross stress, predicted stress, final stress)
% temp-out % \end{itemize}

\section{Mesh Generation using GiD}
\label{Mesh_Generation_using_GiD}

\begin{enumerate}

  \item  Download  the latest version of GiD from 
\url{http://www.gidhome.com/},
   and also get a temporary license (or purchase it...).


   \item  Download
\href{http://sokocalo.engr.ucdavis.edu/~jeremic/lecture_notes_online_material/_Chapter_SoftwareHardware_Domain_Specific_Language_English/essi.gid.tar.gz}
   {essi.gid.tar.gz},
   unpack it  (\texttt{tar  -xvzf  essi.gid.tar.gz})  in  \texttt{problemtypes}
   directory that is located in GiD's root directory.

   \item  When  you run GiD, you will see \texttt{essi} in "Data $>$ Problem types",
   and can start using it...

   \item A simple movie with instructions for mesh generation is available:
\href{http://sokocalo.engr.ucdavis.edu/~jeremic/lecture_notes_online_material/_Chapter_SoftwareHardware_Domain_Specific_Language_English/HowtoMakeEssiInput_byGID.mpg}
   {(Link to a movie, 11MB)}.

 \end{enumerate}
